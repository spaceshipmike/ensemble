## What went well

- **The agents/commands symmetry was the right shape.** Chunk 4 paved
  the road and chunk 5 drove it: identical CRUD surface, identical
  dual-field contract, identical fan-out with the `__ensemble: true`
  frontmatter marker. The one real divergence — hyphenated YAML keys
  (`allowed-tools`, `argument-hint`) for the Claude Code convention —
  fell cleanly into the per-type frontmatter parsers. Considered
  extracting a shared `frontmatter-store.ts` helper mid-chunk-5; the
  duplication is narrow enough that the cost of the abstraction would
  have exceeded the gain. Left separate.

- **The capture-set discipline held.** Extending `syncClient`'s snapshot
  capture to include every `~/.claude/agents/*.md` and
  `~/.claude/commands/*.md` fan-out path was a one-line addition per
  resource type. Rollback now restores agents and commands byte-identical
  — verified end-to-end in `tests/agents-sync.test.ts` and
  `tests/commands-sync.test.ts`.

- **Doctor v2 integrated without disturbing existing checks.** All four
  new findings are pure additions: `checkOrphanSnapshots`,
  `checkSnapshotDirSize`, `checkAgentsCommandsDrift`,
  `checkRetentionConfigVisibility`. Existing tests passed unchanged.
  The `snapshot_dir_size_warn_mb` field slotted into `SettingsSchema`
  with a default that matches the approved assumption (500 MB).

- **The snapshots sub-router was the cleanest possible addition to the
  monolithic `appRouter`.** Three procedures (`list` / `show` /
  `restore`), three mock entries in `router.test.ts`, three new tests.
  The spec-vs-review note calling out the sub-router as a v2.0.1 target
  meant the shape was pre-approved; execution was mechanical.

- **The restore dialog copy literally mirrors the CLI.** The approved
  assumption "restore dialog copy mirrors `ensemble rollback` CLI
  warning" is now structurally enforced in the Playwright spec, which
  asserts on the exact template: "Restore snapshot from {when}? This
  will overwrite N files... n files will be deleted. Continue?" — if
  the CLI warning drifts, either a CLI test or this spec will fail.

- **Zero regressions across 70 new tests.** Library went 306 → 373 with
  no legacy test flipping red; desktop went 28 → 31 similarly. The
  primitives from chunks 1-3 (snapshots, mergeSettings, hooks) absorbed
  chunks 4-7 as planned without surgery.

## What was harder than expected

- **Biome's `useLiteralKeys` bit my new files.** `src/skills.ts` and
  `src/hooks.ts` use `meta["foo"]` bracket notation as an existing
  convention; when I mirrored it in `src/agents.ts` and `src/commands.ts`,
  biome surfaced them as brand-new violations (the existing files are
  grandfathered under the pre-existing-errors baseline). Converted to
  dot notation in my new files. The style inconsistency (new files use
  `meta.foo`, old files use `meta["foo"]`) is a small readability tax —
  but less of a tax than either running `biome check --write` across
  `src/` (retro 1-3 already flagged that as destructive) or shipping
  new files with fresh violations.

- **Import ordering was a small recurring chore.** Biome's
  `organizeImports` rule flagged three files during this build (`doctor.ts`,
  `e2e/snapshots.spec.ts`, and briefly `src/index.ts`). Each fix was a
  five-second reorder — but because I can't run `biome check --write` in
  this codebase, each had to be done manually. Net cost: ~two minutes
  over the build. Noted: for future chunks, draft imports in biome-
  preferred order up front (node: builtins → `@` packages → relative).

- **Column-limit formatter flapping.** Moving a variable name from
  `ensemble-agents-sync-` to `ensemble-cmd-sync-` flipped one line from
  wrapped to one-liner under biome's 100-column rule, which counted as a
  new formatter error. Trivial to fix, mildly annoying. The codebase's
  100-col default is strict — this rule will keep surfacing on tmpDir-
  generating prefixes until the pattern is helper-extracted.

- **The dist rebuild was a timing surprise.** The desktop package
  consumes the library through the workspace `ensemble` import, which
  resolves to `dist/` (not `src/`). When I added `listSnapshots` to
  `src/index.ts` and then typechecked desktop, the import failed
  silently until I ran `npm run build` at the root. This is working
  as designed (desktop should consume a built library, not source), but
  it's a gotcha for any future chunk that adds new library exports
  consumed by desktop in the same session.

## What didn't work

- **Nothing had to be abandoned or rearchitected.** All four chunks
  shipped in their originally planned shape. No chunk required a
  restructure; no experience-level ambiguity surfaced. The three
  pre-resolved assumptions (global-scope default for agents, 500 MB
  snapshot-dir size default, CLI-mirror restore dialog copy) carried
  the build cleanly through — worth investing in that up-front
  decisiveness again on future builds.

## Follow-ups to queue

1. **Spec-parity clean-up.** The CLAUDE.md "Target modules (v2.0.1, not
   yet built)" table lists `agents.ts`, `commands.ts`, and `snapshots.ts`
   as targets. Two of those (agents, commands) are now real modules with
   real tests; `snapshots.ts` has existed since chunk 1. Update CLAUDE.md
   to move those rows from the target table into the shipping table, and
   widen the "additive sync only" rule to cover agents/commands (the
   current rule text says "servers, plugins, skills"; the reality now
   includes agents/commands/hooks/settings).

2. **CLI verbs for agents/commands.** We shipped library + sync + canonical
   store, but `ensemble agents add/list/remove` and `ensemble commands ...`
   do not yet exist in `src/cli/index.ts`. Users can import agents/commands
   programmatically or via the desktop app, but the CLI surface is still
   limited to what the Claude Code agent authors can sync via the
   filesystem. Worth a tight follow-up chunk.

3. **Desktop playwright run.** The `e2e/snapshots.spec.ts` spec is
   written and should pass — but I did not execute the Playwright run in
   this build session (it requires an electron-vite build + real
   launch). Worth a single-command verification pass
   (`cd packages/desktop && npm run build && npm run test:e2e`) before
   merging into a release branch.

4. **Existing `e2e/layout.spec.ts` has a stale assertion.** The test
   "switching to DOCTOR view shows placeholder" expects text "DOCTOR ·
   SOON", which no longer exists in DoctorView. Pre-existing failure;
   not touched by this build, but the next desktop-focused chunk should
   clean it up.

5. **Doctor drift check scope.** The agents/commands drift check only
   runs against clients that have both `agentsDir` / `commandsDir`
   declared — today that's just `claude-code`. When other clients
   (Cursor, Windsurf) gain those directories, the check will cover them
   for free — but a spot-check is worth doing at that time.

6. **Snapshot-dir size walk is recursive but naive.** It stat-walks every
   file under `~/.config/ensemble/snapshots/` every time doctor runs.
   Fine for MB-scale snapshot dirs; if retention is ever set to
   "keep everything" and the directory grows into the GB range, this
   becomes visible in doctor latency. Not urgent; flag for future.

7. **Minor biome style drift in new files.** The dot-notation vs
   bracket-notation split between `src/skills.ts`+`src/hooks.ts`
   (bracket) and `src/agents.ts`+`src/commands.ts` (dot) is cosmetic.
   A single-commit normalisation pass (pick one style, apply uniformly,
   update the biome baseline) would tidy it up without functional
   change.

## Wins

1. **Agents and commands are first-class.** Users can now install,
   enable/disable, and uninstall subagents and slash commands through
   the same library surface that already handled servers, plugins, and
   skills. Fan-out to `~/.claude/agents/` and `~/.claude/commands/`
   respects user-authored files byte-identical (the `__ensemble: true`
   marker is the hinge) and is covered by rollback.

2. **Snapshots are now visible.** Before chunk 7, snapshots existed on
   disk but the only way to browse them was to `ls ~/.config/ensemble/
   snapshots/` and hope the filenames were informative. Now the desktop
   app shows the snapshot ring, per-snapshot metadata, and a per-file
   manifest — with a restore path that double-confirms using the same
   language the CLI uses.

3. **Doctor now speaks safe-apply fluently.** Four new checks turn the
   snapshot/agent/command surfaces from silent infrastructure into
   health-audit citizens: orphan detection, size ceiling, drift vs the
   library, and retention-policy transparency. A user asking "is my
   snapshot dir getting out of hand?" now gets a first-class answer.

4. **Four chunks, four commits, four tags, zero regressions.** The plan
   proposed a clean linear dependency chain (1-3 shipped → 4 → 5 → 6
   → 7) and that's exactly how it ran. No mid-build re-ordering, no
   rollbacks, no experience-level questions surfaced to the user.

5. **v1.0.14 is a coherent release.** The jump from v1.0.10 (post-retro
   chunks 1-3) to v1.0.14 traces exactly to four named features, each
   with its own commit, its own tag, and its own test suite. The
   changelog writes itself.
