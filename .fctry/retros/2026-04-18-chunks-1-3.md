## What went well

- **Snapshots landed as a clean primitive.** The lazy single-capture-per-
  syncClient-call design threaded through every write site (main client
  path, CC plugins, CC hooks, project-level .claude/settings.json) with a
  single closure. No plumbing cascade, no surprise call-sites missed. The
  rollback CLI verb worked first try once the capture surface was right.

- **The capture-files set is correct.** We pre-compute every file that
  syncClient *could* touch (including new-files that will be created) before
  the first real write, so any post-sync rollback is comprehensive. The
  new-file / existing-file distinction in the manifest means rollback
  deletes files the sync created rather than leaving orphans.

- **mergeSettings composed cleanly with hooks fanout in chunk 3.** Because
  chunk 2 shipped the primitive in isolation (with a real syncCCPlugins
  call already routing through it), chunk 3's hooks integration reduced to
  "add `hooks` to the ownedKeys array and include `nextHooks` in the
  managed object." The byte-identity invariant for user keys held across
  both chunks without any special-case plumbing.

- **Test isolation via env vars held up.** `ENSEMBLE_SNAPSHOTS_DIR`,
  `ENSEMBLE_HOOKS_DIR`, and `HOME` overrides sandboxed the integration
  tests fully; no test ever touched real `~/.claude` or real
  `~/.config/ensemble`.

- **Dual-field contract is structurally enforced.** `toSettingsEntry()`
  strips `userNotes` and `description` with `Omit` types and runtime
  extraction; the "userNotes never round-trips" test asserts on the
  literal entry shape; `toStoredShape()` likewise drops `description`
  before writing the canonical JSON. Three layers, one invariant.

## What was harder than expected

- **The `homedir()` caching bug in `CC_SETTINGS_PATH`** was the only real
  setback. The integration test for hooks fanout failed because
  `readCCSettings()` resolved `~/.claude/settings.json` against the
  module-load-time home directory rather than the test's overridden
  `$HOME`. Fix was one `ccSettingsPath()` getter and a two-line swap —
  but diagnosing "why isn't settings.json being written to tmpDir" took a
  couple of minutes of grep. Noted: any path helper that bakes a value at
  module load will fight with sandboxed tests.

- **Biome's `--write` over-reach.** Running `biome check --write` on all
  `src/` applied formatting/import-sort fixes to ~40 files I hadn't
  touched, polluting the chunk 1 diff. Had to `git checkout HEAD --` the
  unrelated files and re-apply chunk 1 edits to keep the commit focused.
  For chunk 2 and 3 I only ran `biome check` on the files I'd actually
  modified. Takeaway: in this codebase, never run `biome check --write`
  across the whole src/ during a chunk — scope it to the changed files.

- **Pre-existing biome diagnostics made "lint must pass" ambiguous.** The
  project's `npm run lint` already reports ~180 pre-existing errors (non-
  null assertions, computed keys, etc.) on files untouched by this build.
  The baseline was not clean. I treated the constraint as "my new code
  must pass biome cleanly and I must not introduce new violations" —
  which all three chunks satisfy. A proper lint cleanup pass is a
  separate out-of-scope concern; flagged below.

## What didn't work

- **Nothing had to be abandoned or rearchitected.** The slim-cut plan
  shipped as designed. No chunk required a restructure or escalate.

5. **`doctor` checks for orphaned snapshots / snapshot-dir size.** The

7. **The next plan should tackle:** chunks 4-7 of the original v2.0.1

## Wins

1. **Rollback is real.** A user can now sync, realise they shouldn't have,
   and type `ensemble rollback --latest` to get back to byte-identical
   pre-sync state. This was arguably the biggest missing safety net in
   Ensemble v1; it's in now.

2. **The byte-identity invariant is a testable contract, not a promise.**
   The critical chunk 2 test constructs a settings.json with 10
   heterogeneous user keys (nested permissions, env, hooks entries,
   numbers, booleans, strings), runs a merge touching 2 owned keys, and
   asserts every user key serialises byte-identical pre/post. No
   hand-waving: the invariant either holds or a test turns red.

3. **The dual-field contract holds across three resource types.**
   `description` (auto-computed, source-owned) and `userNotes` (user-
   authored, library-side only) are now encoded in HookSchema,
   SettingSchema, and the existing ServerSchema / PluginSchema / SkillSchema.
   Re-import preservation tests already covered the first three; the
   hooks test adds the fourth.

4. **Zero test regressions across 47 new tests.** The full suite went
   259 → 306 without a single legacy test flipping red. That's strong
   evidence the new primitives didn't leak into existing behaviour paths.

5. **The `ccSettingsPath()` getter diagnosed one latent testability bug.**
   It was never going to bite a real user (HOME doesn't change at
   runtime) — but catching it inside a chunk whose tests genuinely
   required $HOME isolation was the right moment to fix it.
