## What went well

- **The v2.0.1 noun-first grammar landed without legacy breakage.** Chunk 8
  replaced four top-level / per-type verbs (`enable <server>`,
  `disable <server>`, `plugins install/uninstall/enable/disable`) with the
  unified `pull` / `install` / `uninstall` / `remove` surface — and did so
  behind a new `src/lifecycle.ts` dispatcher that routes to the existing
  operations functions. Zero call-site churn to `operations.ts`; every old
  test stayed green on its original code path. The `ensemble remove <name>`
  verb retained its v1.3 orphan-detection hint even with the type-filter
  upgrade, so users running `ensemble remove <old-server>` see the same
  error message they did in v1.1.x.

- **`src/settings.ts` from chunk 2 absorbed the settings-verb chunk with
  zero surgery.** The primitive (`mergeSettings`, `buildManagedFromList`,
  `readOwnedKeys`) landed two chunks earlier with no callers beyond the
  sync engine. Chunk 8's `ensemble settings sync` composes those three
  functions into four lines. The `#settings` scenario (L1879) — deferred
  from the post-chunks-1-3 review exactly because the CLI wasn't ready —
  passes on the first write.

- **The browse engine didn't need fuzzysort.** The approved plan mentioned
  fuzzysort as a candidate dependency, but a 10-line subsequence scorer
  (gap + length + start-position penalties, lowercase-normalised) gave
  correct ranking on every test case. Keeping `browse.ts` dependency-free
  matched the v2.4.0 evolve mandate ("pure-function library primitive")
  and removed one npm install from the desktop build. Considered adding
  fuzzysort as an optional peer dep for richer matching; the cost
  (dependency surface, one more supply-chain link) exceeded the marginal
  ranking quality.

- **Library barrel exports stayed in sync.** Adding `browse.ts`,
  `lifecycle.ts`, and `managed-settings.ts` to `src/index.ts` meant the
  desktop package (`ensemble: file:../..`) picked them up after a single
  `npm run build`. No re-export surgery, no unused type leaks — the
  dist-based consumption model worked.

- **Sub-router symmetry held.** Chunk 10's four new routers
  (`agentsRouter` / `commandsRouter` / `hooksRouter` / `settingsRouter`)
  all mirror the `snapshotsRouter` shape: zod-validated input, one
  `fresh()` read per mutation, `saveConfig()` on success, no magic. The
  diff in `router.ts` is mechanical enough that the +12 contract tests
  were template-copies with different mock values. The temptation to
  abstract a "resource-router factory" was resisted; the duplication is
  narrow and readable.

- **Per-chunk version bumps are visible in the git log.** Auto-version
  hook handled v1.2.1 / v1.2.2 / v1.2.3 transitions cleanly; each chunk
  commit has a paired tag. The public-surface break (1.1.x → 1.2.x) is
  unambiguous in `git tag --sort=-creatordate`.

## What was harder than expected

- **`require()` calls in action handlers don't work outside the project
  build.** The existing `src/cli/index.ts` had several `require("../foo.js")
  as typeof import("../foo.js")` inline requires inside commander action
  handlers — a pattern that latently fails under ESM (`ReferenceError:
  require is not defined`). The original file worked because the pre-
  existing tests never exercised those paths. My first pass at chunk 8
  carried the pattern forward for the new `settings` verbs; tests promptly
  surfaced the bug. Fix: replaced every new `require()` with top-level ES
  imports. Left the legacy `require()` calls alone — they're still latent
  bugs, but out of scope for chunk 8. Logged as a deferred insight for a
  future review: convert the remaining 5–6 `require()` calls in `cli/index.ts`
  to ES imports.

- **The auto-version hook doesn't distinguish minor-bumps from patch-
  bumps.** The chunk 8 plan called for a manual `1.1.2 → 1.2.0` minor bump
  (public-surface break). I hand-edited `package.json` and
  `.fctry/config.json` to `1.2.0`; the auto-version `PostToolUse` hook on
  `git commit` then auto-incremented to `1.2.1`. So chunk 8 shipped as
  `v1.2.1`, not `v1.2.0`. The actual user-facing effect (minor-version jump
  from 1.1 to 1.2) still landed — just with an extra patch bump — and
  chunks 9 and 10 shifted to v1.2.2 and v1.2.3 accordingly. Logged as a
  follow-up: consider a `--bump minor` flag on the hook or a way to suppress
  the auto-patch when the commit message includes "major:" / "minor:"
  markers.

- **Latent pre-existing biome violations surfaced under `biome check`.**
  Running `biome check src/cli/index.ts` reports ~7 pre-existing violations
  on untouched code (template literals, `useLiteralKeys`, `parseInt` →
  `Number.parseInt`, import organization). The chunks 1-3 retro explicitly
  banned `biome check --write src/` across the tree; I obeyed and manually
  fixed only the four issues introduced by my own edits. Running
  `git stash && biome check && git stash pop && biome check` showed the
  count dropping from 7 → 5 after my chunk-8 changes (I removed code with
  pre-existing violations). Net improvement, zero new violations.

- **Playwright Electron launch still won't fire locally.** The chunks 4-7
  retro flagged this; I tried again at the chunk 10 end and got the same
  process-launch failure on `e2e/snapshots.spec.ts`. The failure is in
  test setup, not the tests themselves. Logged as a known-blocked
  follow-up.

- **`browseSearch` vs. `enableSkill` dual-import collision in the CLI.**
  Chunk 9's barrel export of `browseSearch` meant I needed to import it
  into `cli/index.ts` alongside the existing `enableSkill` / `enableAgent`
  operations. TypeScript was happy once I wired the aliased imports
  correctly, but my first pass tried to re-import `enableSkill` with an
  alias, which collided with the already-imported name from operations.
  Renamed on the fly; cost a minute of debugging.

## What didn't work

- **The lifecycle `pull` verb can't actually clone git sources yet.** The
  spec lists `owner/repo` and URL forms as routing through "existing add
  paths," but the existing paths stop at marketplace registration — there's
  no `git clone` → local directory handoff in the v1.x operations. Chunk
  8's `pull` routes both owner/repo and URL to `addMarketplace` (which is
  the truthful answer for a plugin catalog) but doesn't try to clone the
  repo. A future chunk will need to add a `pull-and-extract` flow for
  local-directory resources fetched from git. Logged as a deferred insight
  in this retro rather than surfaced as a mid-build ambiguity because the
  scenario didn't depend on it.

- **The `registry:<slug>` form is a hint, not a fetch.** Same reason: the
  existing `registry add` is async and I didn't want to tangle the
  lifecycle dispatcher with async registry I/O in chunk 8. The `pull`
  verb returns an instruction ("Run `ensemble registry add <slug>`")
  rather than doing the fetch itself. This is correct for the v2.0.1
  cut but a future chunk should thread the async fetch into `pull`
  directly for UX parity with the other three source forms.

## Follow-ups

1. **Convert remaining `require()` calls in `cli/index.ts` to ES imports.**
   Latent bug; affects five or six code paths (`findOrphanedInClients`,
   `clearUsage`, `loadUsage`, `exportGroupAsPlugin`, `mkdtempSync`-style
   inline requires, the `rollback` command helper). Individually mechanical;
   collectively a tiny cleanup chunk.

2. **Pull `git clone` support into the lifecycle dispatcher.** The v2.0.1
   surface promises four source forms for `pull`; two of them currently
   degrade to marketplace-add or a text hint. A follow-up chunk should
   add an async `pull` path that clones a local working copy for
   owner/repo + URL plus directory-pull for local skills/plugins.

3. **Consider a commit-message-driven minor-bump flag on the auto-version
   hook.** If a commit message starts with `chunk N:` or contains
   `public-surface break`, the hook could bump minor instead of patch.
   Would avoid the chunk-8 version-number drift documented above.

4. **Fix Playwright Electron launch.** This is the third retro that names
   this issue; it's not going away on its own. A dedicated investigation
   chunk would unblock the snapshots e2e spec and the layout spec's
   MATRIX / DOCTOR assertions.

5. **Expose the four new sub-routers in the renderer.** Chunk 10 shipped
   the tRPC surface but the renderer still only wires Doctor / Matrix /
   Patch / Snapshots / Registry. Adding Agents / Commands / Hooks /
   Settings panels is a UI-chunk that consumes the already-shipped
   contract.

## Wins

- **The CLI grammar is cleaner than it was.** Users can now say
  `ensemble pull acme/some-marketplace`, then `ensemble install server-x`,
  then `ensemble uninstall server-x`, then `ensemble remove server-x`.
  The verbs do what their English names say. `ensemble settings set
  permissions.allow '["Read","Grep"]' --client claude-code` does the
  long-deferred `#settings` scenario in one line.

- **The browse engine is the smallest possible thing that works.** ~120
  lines of pure-function TypeScript, zero dependencies, deterministic
  ranking, passes all three `#browse-engine` scenarios. Both the CLI
  and the desktop Registry consume the same function — identical ranking
  on identical input, verified by a test that runs `browseSearch` twice
  and asserts the results are `.toEqual`.

- **Desktop IPC is now complete.** Every v2.0.1 resource type has a typed
  tRPC procedure: agents, commands, hooks, skills, servers, plugins,
  managed settings, snapshots, marketplaces, and the browse engine.
  Chunk 10's +12 contract tests document the full surface.

- **Three chunks, three commits, three version tags, zero rebase.** Clean
  linear history. The auto-version hook applied patch bumps on each
  commit; biome + tsc gates were honoured; the test suite stayed green
  from 379 → 430 in exactly the expected size per chunk.
